# firebase

## Purpose
Firebase project configuration, security rules, Firestore indexes, seed data for local and production environments, and Firebase Cloud Functions. This directory is the deployment boundary for all Firebase-managed infrastructure: Firestore, Firebase Storage, and Cloud Functions.

## Key Files
| File | Description |
|------|-------------|
| `firebase.json` | Firebase project configuration — emulator ports, functions runtime, hosting targets |
| `.firebaserc` | Project aliases (`default` → dev project, `prod` → prod project) |
| `firestore.rules` | Production Firestore security rules |
| `firestore-test.rules` | Relaxed rules used during local emulator testing |
| `firestore.indexes.json` | Composite Firestore index definitions |
| `storage.rules` | Firebase Storage security rules (public read for `public/**`) |
| `firestore_data/dev/parties.json` | Dev seed data for parties collection |
| `firestore_data/dev/candidates.json` | Dev seed data for candidates collection |
| `firestore_data/dev/municipalities.json` | Dev seed data for municipalities collection (generated by `municipalities_sync.py`) |
| `firestore_data/dev/election_types.json` | Dev seed data for election types collection |
| `firestore_data/dev/proposed_questions.json` | Dev seed data for proposed questions |
| `firestore_data/prod/parties.json` | Production parties seed data |
| `storage_data/public/NOTE.md` | Notes on Firebase Storage public directory structure |

## Subdirectories
| Directory | Purpose |
|-----------|---------|
| `functions/` | Firebase Cloud Functions for PDF ingestion (see `functions/AGENTS.md`) |

## For AI Agents

### Working In This Directory
- Deploy rules and indexes with: `firebase deploy --only firestore:rules,firestore:indexes,storage` (from the `firebase/` directory or repo root with `--project` flag)
- Use `firebase.json` emulator configuration for local development; default ports: Firestore 8081, Storage 9199, Functions 5001
- Seed data JSON files follow the format expected by `scripts/import-firestore.js`; each file is an array of objects with an `id` field and the document data
- `firestore_data/dev/municipalities.json` is auto-generated by `src/services/municipalities_sync.py` — do not edit by hand

### Testing Requirements
```bash
# Start Firebase emulators (from firebase/ directory)
firebase emulators:start

# Import seed data
node ../scripts/import-firestore.js parties firestore_data/dev/parties.json
node ../scripts/import-firestore.js candidates firestore_data/dev/candidates.json
```

### Common Patterns
- Firestore collection paths used by the application: `parties`, `candidates`, `proposed_questions/{party_id}/questions`, `cached_answers/{party_id}/{cache_key}`, `system_status/llm_status`, `sources/{party_id}/source_documents`, `municipalities`, `election_types`
- Storage layout: `public/<party-subdir>/<document_name>_<YYYY-MM-DD>.pdf` — Cloud Functions trigger on upload to `public/**`
- Rules use `request.auth != null` for write operations; public read is allowed on `parties`, `candidates`, and `proposed_questions`
